# ADR-012 Enriched Output Design

- Status: accepted
- First needed: start of P7

## Context

flatline needed a richer structured output surface for downstream graph and
data-flow analysis without inflating the baseline decompilation contract or
leaking native handles across the ABI.

## Decision

Make enriched output opt-in through `DecompileRequest.enriched`. Extract pcode
and varnode data after `Action::perform()` from the post-simplification
`Funcdata`, and surface it as `DecompileResult.enriched` with a dedicated
`Enriched.pcode` layer instead of expanding `FunctionInfo`. Represent pcode ops
and varnodes with stable integer IDs, string opcode names, string address-space
names, and ID-based graph edges. `Pcode.to_graph()` projects the data into a
caller-owned `networkx.MultiDiGraph`.

## Consequences

- The baseline result stays lightweight when enriched output is not requested.
- Downstream analysis gets stable, Python-owned graph data with no native
  pointers crossing the ABI.
- If a caller explicitly requests enriched output, flatline must either provide
  it or fail rather than silently downgrade.
