# ADR-001 Public Scope Model

- Status: accepted
- First needed: end of P0

## Context

flatline needed a durable MVP input model. The tradeoff was between a direct
memory-and-architecture contract, a binary-loading contract, or a hybrid that
treated binary loading as part of the baseline public API.

## Decision

Adopt memory image + base address + target selection + function address as the
core public contract. File-to-memory convenience is deferred to a later,
additive layer.

## Consequences

- `DecompileRequest` keeps `memory_image` and `base_address` in the baseline
  contract.
- The foundational workflow remains single-function decompilation from
  caller-provided memory.
- Fixture strategy uses raw memory images rather than binary files as runtime
  test inputs.
- Any future binary-loading convenience must stay additive over the memory
  model instead of redefining it.
